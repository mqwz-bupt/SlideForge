## 一、项目概述与选题（Q1-Q4）

### Q1：请用一分钟介绍你们的项目 SlideForge。

**答：** SlideForge 是一个基于 Electron 的 AI 桌面应用，核心功能是将文档（PDF、DOCX、TXT、Markdown）自动转化为精美的演示文稿。用户上传文档后，系统通过 AI 解析文档结构，生成大纲，然后用户选择视觉风格，系统自动生成包含动画和装饰效果的幻灯片，最后可以导出为 HTML、PPTX 或 PDF。我们支持 5 家国内主流 AI 供应商（DeepSeek、通义千问、智谱 GLM、百度文心、OpenAI），内置 16 套完整的视觉主题，13 种幻灯片布局。整个流程从上传到导出控制在 3 分钟以内。

### Q2：为什么选择这个选题？目标用户是谁？

**答：** 选题来源于真实痛点。中国大学生每学期平均制作 3-5 次 PPT，但大多数人缺乏设计能力，花费大量时间在排版上而非内容上。现有的 AI PPT 工具（如 Gamma、Tome）都是 SaaS 产品，存在数据隐私问题和网络依赖。我们选择 Electron 桌面应用架构，所有数据本地存储（SQLite），文档不上传到任何服务器，只有 AI 文本交互经过 API。目标用户是大学生、教师和职场人士，尤其是需要频繁制作 PPT 但缺乏设计经验的人群。

### Q3：你们的项目和 Markitdown + Banana-slides 有什么区别？

**答：** 这是一个很好的问题。Markitdown 是微软的 Markdown 转换工具，Banana-slides 是一个浏览器端的幻灯片展示库。它们都是单一功能的开源工具。SlideForge 的差异在于三个层面：

**第一，全链路整合。** 我们把文档解析、AI 大纲生成、风格选择、幻灯片渲染、AI 对话优化、多格式导出整合成一个完整的桌面工作流，用户不需要在多个工具之间切换。

**第二，多模型适配器架构。** 我们设计了一个统一的 AI 适配器模式，支持 5 家 AI 供应商的热切换。用户可以配置自己的 API Key，选择最适合自己的模型，而不是被锁定在一个供应商上。

**第三，设计 Token 系统。** 我们不是简单的颜色替换，而是为每个主题定义了完整的设计系统——包括颜色、字体排版体系（Typography Scale）、间距、圆角、组件样式、装饰元素策略和情绪元数据。这让我们可以做到智能主题推荐。

### Q4：你们参考了 zarazhangrui/frontend-slides 项目，借鉴了什么？自己的增量在哪里？

**答：** frontend-slides 是一个 Claude Code Skill，它提供了幻灯片样式预设的设计思路和 CSS 实现参考。我们从它借鉴了"设计 Token"的理念和部分样式方向。但我们的增量是显著的：

1. **架构层面**：frontend-slides 是一个静态 HTML 生成器，SlideForge 是完整的桌面应用（Electron + React + Vite），有 IPC 通信、状态管理（Zustand）、数据持久化（SQLite）
2. **AI 能力**：frontend-slides 没有 AI 集成，我们有完整的 AI 工作流——文档解析、大纲生成、幻灯片生成、对话式优化
3. **设计系统深度**：我们把 frontend-slides 的平面 CSS 配置升级为结构化的 DesignTokens 类型系统，包含 CJK 字体配对、情绪匹配元数据、组件模式定义
4. **导出能力**：frontend-slides 只输出 HTML，我们支持 HTML/PPTX/PDF 三种格式
5. **用户体验**：我们有完整的 Wizard 引导流程、大纲编辑器、AI 聊天面板，而不是命令行工具

## 二、技术架构（Q5-Q10）

### Q5：你们的整体技术架构是什么？为什么这样选？

**答：** 我们的技术栈是 **Electron + React 18 + Vite + Emotion CSS-in-JS**。

| 层 | 技术选择 | 理由 |
|---|---|---|
| 桌面框架 | Electron | 跨平台、文件系统访问、本地数据存储 |
| 前端框架 | React 18 | 组件化、生态成熟、hooks 适合复杂状态 |
| 构建工具 | Vite (electron-vite) | 极快的 HMR、原生 ESM 支持 |
| 样式方案 | Emotion CSS-in-JS | 主题系统、动态样式、与 React 深度集成 |
| 状态管理 | Zustand | 轻量、TypeScript 友好、无 boilerplate |
| 数据存储 | better-sqlite3 | 本地优先、零网络依赖、同步 API 性能好 |
| 国际化 | i18next | 成熟的 i18n 方案、中英双语支持 |

选择桌面应用而非 Web 应用，核心原因是**数据隐私**——用户的文档和 API Key 都保存在本地，API Key 还通过 Electron 的 safeStorage 进行加密存储。

### Q6：你们的主进程和渲染进程是怎么通信的？

**答：** 使用 Electron 的 IPC（进程间通信）机制。主进程负责文件系统操作、AI API 调用和导出功能，渲染进程负责 UI 展示和用户交互。

我们定义了一组 contextBridge 暴露的 API：
- `file:open-text` — 读取文档文件
- `file:open-image` — 读取图片
- `ai:chat` / `ai:chat-stream` — AI 对话（支持流式响应）
- `ai:test-key` — 测试 API Key 有效性
- `export:html` / `export:pptx` / `export:pdf` — 三种格式导出
- `project:save` / `project:load` / `project:list` — 项目持久化
- `safe-store:get` / `safe-store:set` — 安全存储（用于 API Key 加密）

渲染进程通过 `window.api.invoke()` 调用这些接口，所有数据通过序列化传输，保证了进程隔离的安全性。

### Q7：AI 适配器是怎么设计的？支持哪些模型？

**答：** 我们设计了一个**统一适配器模式（Adapter Pattern）**：

```typescript
interface AIProviderConfig {
  provider: 'deepseek' | 'qwen' | 'glm' | 'openai' | 'wenxin'
  apiKey: string
  model: string
  baseURL: string
}
```

每个供应商有一个预设配置（Provider Preset），包含 `baseURL` 和默认 `model`。因为国内的 AI 供应商（DeepSeek、通义千问、智谱、百度文心）都兼容 OpenAI 的 Chat Completion API 格式，所以我们可以用统一的调用接口，只需要切换 `baseURL` 和 `model` 就能切换供应商。

用户在设置中配置 API Key，选择供应商，系统自动填充对应的 base URL 和默认模型。这样新增一个供应商只需要添加一个 preset 配置，不需要写新的适配器代码。

### Q8：你们的数据存储方案是什么？为什么不用云数据库？

**答：** 我们使用 **better-sqlite3**（SQLite 的 Node.js 绑定）作为本地数据库。选择理由：

1. **隐私优先**：用户文档可能包含学术论文、商业计划等敏感内容，不应上传到云端
2. **零配置**：SQLite 是嵌入式数据库，不需要安装数据库服务，用户下载即用
3. **性能**：better-sqlite3 是同步 API，避免了异步数据库操作的复杂性，在 Electron 主进程中性能表现好
4. **离线可用**：除了 AI 调用需要网络外，所有功能都可以离线使用

项目数据包括项目元信息、大纲结构、幻灯片数据都存在 SQLite 中，API Key 通过 Electron 的 safeStorage 加密后存储。

### Q9：你们的 16 个主题是怎么实现的？技术细节是什么？

**答：** 我们设计了一个**双层设计 Token 系统**：

**渲染层（DesignTokens）**：定义在 `stylePresets.ts` 中，每个主题是一个完整的 DesignTokens 对象，包含：
- 颜色系统（ColorTokens）：12 个颜色变量（bg、primary、text、textMuted、accentLight 等）
- 字体排版体系（TypographyTokens）：9 个排版级别（h1、h2、h3、body、metricValue 等），每个包含 fontFamily、fontWeight、fontSize（使用 clamp 响应式）、lineHeight、letterSpacing
- 间距系统（SpacingTokens）、圆角系统（RadiiTokens）
- 组件模式（ComponentTokens）：卡片、标签、强调线、封面装饰的具体样式
- 装饰策略（DecorativeStyle）：9 种装饰风格
- 情绪元数据（ThemeMetadata）：用于智能推荐

**导出层（StyleConfig）**：定义在 `styles.ts` 中，用于 HTML/PPTX 导出，包含颜色变量、Google Fonts URL、背景特效和主题专属 CSS。

两层通过 `tokensToLegacyConfig()` 函数自动同步，确保渲染预览和导出结果一致。

### Q10：你的幻灯片布局系统是怎样的？有几种布局？

**答：** 我们有 **13 种幻灯片布局**：

| 布局 | 用途 | 占比控制 |
|------|------|----------|
| title | 封面/标题页 | — |
| content | 正文内容 | 上限 15% |
| two-column | 双栏对比 | 最低 20% |
| feature-grid | 特性网格 | — |
| quote | 引用/金句 | — |
| section-divider | 章节分隔 | — |
| highlight | 重点强调 | — |
| image | 全屏图片 | — |
| big-number | 数据展示 | — |
| timeline | 时间线 | — |
| callout | 提示/注释 | — |
| statement | 全屏声明 | — |
| blank | 空白页 | — |

在 AI 生成幻灯片时，我们的 prompt 中嵌入了**布局多样性规则**：content 布局不超过 15%，two-column 至少 20%，禁止连续两个相同布局。这确保生成的幻灯片视觉节奏感强，不会全是文字堆砌。

## 三、AI 集成与智能功能（Q11-Q16）

### Q11：AI 在你的系统中具体做了哪些事？

**答：** AI 在 SlideForge 中承担四个核心任务：

1. **文档结构解析**：用户上传文档后，AI 提取文档的章节结构、要点、层级关系，生成结构化大纲（sections 和 points）
2. **幻灯片内容生成**：根据大纲和用户选择的风格，AI 为每一页幻灯片生成标题、正文、布局类型、配图建议等结构化 JSON 数据
3. **对话式优化**：用户可以在 AI 聊天面板中用自然语言修改幻灯片，例如"把第 3 页精简一下"、"给第 5 页加一些数据支撑"，AI 会解析指令并直接修改幻灯片内容
4. **流式响应**：AI 回复使用流式传输（SSE），用户可以看到逐字生成过程，体验流畅

此外，我们还有 5 个快速操作按钮：总结、精简、学术化、补充例子、解释说明，这些是预设的高质量 prompt 模板。

### Q12：你用的什么 prompt 来生成幻灯片？怎么保证 AI 输出正确的 JSON？

**答：** 我们的 prompt 工程分几个层面：

**Prompt 结构**：
- System Prompt 定义了 AI 的角色（专业的演示文稿设计师）和输出格式要求
- User Prompt 包含大纲内容、目标页数、风格要求、布局多样性规则
- 我们在 prompt 中明确列出了 13 种合法的布局类型及其使用场景

**JSON 保障机制**：
- prompt 中明确要求输出 JSON 格式，并给出 schema 示例
- AI 返回结果经过我们自研的 `repairJSON` 流程处理：
  1. `extractJSON` — 从 markdown 代码块中提取 JSON
  2. `repairJSON` — 修复尾逗号、未引用的 key/value、中文引号等常见问题
  3. `fixMissingCommas` — 补充缺失的逗号
  4. `balanceJSON` — 平衡不匹配的括号
- 修复失败时自动降级为 section-by-section 逐段生成（更小的 token 请求，更稳定的输出）
- 最终还有 fallback 机制，生成通用模板幻灯片保证用户始终能看到结果

### Q13：如果 AI 服务挂了或者返回错误，你们的系统会怎样？

**答：** 我们有三层容错机制：

1. **模型切换**：用户配置了多个 AI 供应商，如果一个挂了可以立即切换到另一个，因为我们有统一的适配器接口
2. **分段重试**：如果整批生成失败，系统自动切换为 section-by-section 模式，逐段生成幻灯片，每段请求的 token 数更小，成功率更高
3. **Fallback 兜底**：如果所有 AI 调用都失败，系统会根据大纲自动生成基础模板幻灯片，保证用户至少能看到一个可用的演示文稿，而不是白屏

此外，AI 聊天面板有错误提示机制，流式响应中断时会显示友好的错误消息。

### Q14：你怎么处理 PDF 中的图片、表格和公式？

**答：** 坦率说，这是当前版本的一个已知局限。

我们的 PDF 解析使用 `pdf-parse` 库，它主要提取文本内容。对于：
- **纯文本 PDF**：解析效果良好
- **扫描件/图片 PDF**：pdf-parse 只能提取 OCR 文本（如果有的话）
- **表格**：会丢失结构，变成平铺文本
- **数学公式**：会变成 Unicode 字符或纯文本

对于 DOCX，我们使用 `mammoth.js`，它能较好地保留标题层级和段落结构。

在答辩演示中，我们会使用格式良好的文本型 PDF 或 Markdown 文件来展示最佳效果。未来版本计划集成 OCR 和结构化表格提取。

### Q15：AI 聊天面板是怎么修改幻灯片的？技术实现是什么？

**答：** AI 聊天面板的核心实现：

1. 用户输入指令，发送到 AI（带上当前幻灯片的上下文信息：页码、布局类型、内容摘要）
2. AI 返回包含 `MODIFY_SLIDE` 标记的结构化指令
3. `parseSlideUpdates` 函数解析 AI 返回的修改指令
4. `updateSlideContent` 函数将修改应用到 Zustand store 中对应的幻灯片
5. React 响应式更新自动刷新 UI

支持修改的字段包括：title、body、layout、highlight、accent 等所有 SlideContent 字段。修改完成后会显示"已修改 X 张幻灯片"的反馈。

### Q16：你们的 prompt 是怎么优化的？有没有做 prompt engineering？

**答：** 我们的 prompt 经过多轮迭代优化：

1. **结构化输出约束**：在 system prompt 中明确定义 JSON schema，包含字段类型和枚举值约束
2. **布局多样性注入**：在 prompt 中嵌入布局比例规则，防止 AI 全部生成 content 布局
3. **Few-shot 示例**：提供 2-3 个高质量的输出示例，让 AI 理解期望的输出格式和内容密度
4. **分批生成策略**：先生成大纲摘要（batch 1），再逐段生成详细内容（batch 2），避免 token 限制导致截断
5. **中文优化**：prompt 使用中英双语，针对中文内容的排版特点（如行高、字体配对）做了专门指导

## 四、视觉设计与主题系统（Q17-Q20）

### Q17：你们的 16 个主题是怎么设计的？设计理念是什么？

**答：** 我们的 16 个主题分为四组：

**暗色系（6个）**：Bold Signal、Creative Voltage、Dark Botanical、Cinematic、Aurora、Neon Cyber
**亮色系（5个）**：Notebook Tabs、Pastel Geometry、Split Pastel、Vintage Editorial、Neo Brutalist
**特色系（2个）**：Terminal Green、Swiss Modern
**经典系（2个）**：Swiss Modern、Paper & Ink

设计理念是"**每个主题都是一个完整的设计系统，不是颜色替换**"。每个主题有：
- 独特的字体配对（display font + body font，如 Playfair Display + Inter）
- 完整的 12 色调色板（不是单一强调色）
- 响应式排版体系（使用 CSS clamp() 函数）
- 专属装饰元素（电影 Letterbox、水墨印章、极光光晕等）
- 5 种背景特效（噪点纹理、网格、渐变、圆点、斜线）

新增的 4 个艺术风格主题（Chinese Ink、Cinematic、Neo Brutalist、Aurora）特别回应了"中国风格 vs 好莱坞风格"的文化辨识度需求。

### Q18：Chinese Ink（水墨风）主题的设计细节是什么？

**答：** Chinese Ink 是我们最有文化特色的一个主题：

- **颜色**：宣纸色背景（#f7f3e9）、墨色文字（#2c2c2c）、朱砂印红强调色（#c23a2b）
- **字体**：Noto Serif SC（标题，衬线体，900 字重）+ Noto Sans SC（正文，400 字重），经典中文衬线/无衬线搭配
- **装饰**：印章元素（右下角的方形篆刻）、墨晕效果（大面积模糊圆）、毛笔横线（渐变淡出的分隔线）
- **背景**：噪点纹理（noise-overlay）模拟宣纸质感
- **圆角**：全部为 2px，呼应中国古典建筑的方正线条
- **排版**：更大的内边距（5vw）和行高（1.7），留白充足，符合中国画"计白当黑"的美学

### Q19：你们怎么保证幻灯片在不同屏幕上看起来都正常？

**答：** 三个核心技术保障：

1. **视口锁定**：每张幻灯片强制 `height: 100vh; overflow: hidden`，不会出现滚动条
2. **响应式排版**：所有字体大小使用 `clamp()` 函数，如 `clamp(42px, 5vw, 64px)`，在不同屏幕尺寸下自动缩放
3. **内容溢出保护**：如果内容超出幻灯片容量，prompt 中指导 AI 自动拆分为多页

CSS 变量系统确保主题切换时所有尺寸比例保持一致。

### Q20：你们的"Show Don't Tell"风格选择是怎么实现的？

**答：** 用户在 Wizard 第 3 步选择风格时，我们不是显示文字描述，而是直接**渲染真实的幻灯片预览卡片**：

- 每个主题卡片使用该主题的真实背景色、字体、强调色渲染一个迷你幻灯片
- 预览中显示用户的实际主题文字（如"深度学习技术综述"）
- 用户可以通过情绪标签（confident/excited/calm/inspired）筛选推荐主题
- 选中的卡片有视觉反馈（边框高亮 + 勾选动画）

这就是"Show Don't Tell"——用户看到的是实际效果，不是抽象描述。

## 五、导出与工程实践（Q21-Q25）

### Q21：HTML 导出为什么是零依赖的？怎么做到的？

**答：** 我们的 HTML 导出是一个完全自包含的单文件：

- **所有 CSS 内联**：不引用外部样式表，所有样式写在 `<style>` 标签中
- **所有 JS 内联**：导航、动画、键盘控制写在 `<script>` 标签中
- **Google Fonts CDN**：这是唯一的外部依赖，用于加载字体。如果用户离线，浏览器会 fallback 到系统字体
- **背景特效纯 CSS**：使用 `::before` 伪元素和 CSS gradient 实现，不需要图片资源
- **动画纯 CSS**：使用 `@keyframes` 和 `animation` 属性，不需要 JS 动画库

这样导出的 HTML 文件可以直接双击在浏览器中打开演示，不需要任何服务器或网络。

### Q22：PPTX 导出是怎么实现的？有什么限制？

**答：** 使用 `PptxGenJS` 库生成 `.pptx` 文件。

实现方式：
1. 从 Zustand store 中读取幻灯片数据
2. 获取 StyleConfig 中的颜色和字体配置
3. 根据布局类型（title、content、two-column 等）创建对应的 PPTX 元素
4. 处理颜色值（从 CSS 格式转换为 PptxGenJS 的 hex 格式）

**限制**：
- CSS 特有的效果（如 backdrop-filter、gradient mesh、clip-path）无法完全还原到 PPTX
- 动画效果无法导出到 PPTX（PPTX 的动画是不同的机制）
- 装饰元素（如墨晕、光晕）在 PPTX 中用简单形状近似

HTML 导出是保真度最高的格式，PPTX 导出是功能性的。

### Q23：PDF 导出是怎么做的？

**答：** 使用 Playwright 无头浏览器截图方案：

1. 先生成完整的 HTML 演示文稿
2. 启动 Playwright Chromium 实例
3. 逐页打开幻灯片，等待字体和动画加载完成
4. 对每页进行全页截图（fullPage screenshot）
5. 将所有截图合成 PDF 文件

这种方案的优势是**所见即所得**——PDF 输出和浏览器中看到的完全一致，包括 Google Fonts、CSS 动画最终帧、背景特效等。

### Q24：你们的测试策略是什么？覆盖率如何？

**答：** 我们的测试体系：

**单元测试（104 个测试，6 个测试文件）**：
- `styles.test.ts` — 验证所有 16 个主题的配置完整性、hex 颜色有效性、字体中文字体回退
- `htmlLayouts.test.ts` — 验证 13 种布局的 HTML 生成正确性、主题应用、视口 CSS、动画、导航
- `aiTypes.test.ts` — 验证 5 个 AI 供应商预设的 base URL 唯一性、必填字段完整性
- `docParsing.test.ts` — 验证文档解析路由（按扩展名分发到对应解析器）
- 其他工具函数测试

**测试重点**：我们优先测试了"数据正确性"——主题配置是否完整、AI 类型是否正确、HTML 输出是否包含必需元素。这些是最可能在 AI 生成或主题切换时出问题的环节。

### Q25：你们的代码组织是怎么做的？有多少行代码？

**答：** 项目按功能模块组织：

```
src/
├── main/           # Electron 主进程
│   ├── index.ts    # IPC 处理器、文件操作、窗口管理
│   ├── ai/         # AI 适配器、类型定义
│   └── export/     # HTML/PPTX/PDF 导出、样式配置
├── renderer/       # React 渲染进程
│   └── src/
│       ├── features/   # 功能模块
│       │   ├── wizard/  # 创建向导（5步流程）
│       │   └── editor/  # 编辑器（幻灯片渲染、AI聊天、预览）
│       ├── shared/     # 共享类型、工具、状态管理
│       └── components/ # 通用 UI 组件
└── resources/       # 应用图标等静态资源
```

关键设计原则：
- **features 优先**：按业务功能（wizard、editor）组织，而非按文件类型
- **shared 抽象**：类型定义、工具函数、store 放在 shared 目录，避免循环依赖
- **主进程/渲染进程严格分离**：两个独立的 bundle，通过 IPC 通信

## 六、创新性、难点与团队（Q26-Q30）

### Q26：开发过程中最难的技术问题是什么？怎么解决的？

**答：** 我认为有两个最难的问题：

**第一，AI 输出的不确定性。** AI 模型不保证返回格式正确的 JSON。我们花了很多时间构建 JSON 修复管线：处理尾逗号、未引用的 key、中文引号（""""）、缺失逗号、括号不匹配等问题。最终形成了一个 4 步修复流程（extract、repair、fixCommas、balance），将 AI 输出的 JSON 可用率从约 70% 提升到接近 98%。

**第二，双层设计系统的同步。** 渲染进程的 DesignTokens 和主进程的 StyleConfig 是两个独立的类型系统，但必须保持一致。我们通过 `tokensToLegacyConfig()` 自动转换函数解决了这个问题——渲染层是 source of truth，导出层自动派生，避免手动维护两份配置导致的不一致。

### Q27：你们用了 AI 辅助编程（结对编程），具体是怎么协作的？

**答：** 是的，我们大量使用了 Claude Code（Anthropic 的 AI 编程助手）进行人机结对编程。具体方式：

1. **架构设计**：人类提出需求和技术方向，AI 帮助分析代码库、搜索已有模式、生成实现方案
2. **代码编写**：AI 根据人类审定的 plan 编写代码，人类负责 review 和决策
3. **测试驱动**：AI 帮助编写单元测试，确保主题配置完整性、HTML 输出正确性
4. **代码审查**：使用 code-reviewer agent 对代码进行安全审查和质量检查
5. **Bug 修复**：人类描述 bug 现象，AI 定位根因并修复

这种模式下，人类的角色是**架构决策者**和**质量把关者**，AI 是**高效执行者**。我们的经验是：AI 最擅长的是模式化的实现（如添加新主题的完整配置）、测试编写和代码搜索，但在架构选型和用户体验决策上仍需要人类的判断力。

### Q28：如果让你继续开发，你会优先做什么？

**答：** 按优先级排序：

1. **PPT/PPTX 导入**：支持将现有 PPT 导入并重新设计，这是用户反馈最多的需求
2. **图片智能配图**：集成免费的图片 API（如 Unsplash/Pexels），根据内容自动配图
3. **自定义主题编辑器**：让用户可以调整颜色、字体，生成自己的主题
4. **协作功能**：局域网内的多人协作编辑
5. **更多文档格式**：支持 Excel、PPT 导入解析

### Q29：你觉得你们项目最大的创新点是什么？

**答：** 三个核心创新点：

1. **多模型统一适配器**：5 家 AI 供应商的热切换，国内同类工具很少做到这种程度的模型中立性
2. **结构化设计 Token 系统**：16 个完整的设计系统，包含情绪匹配和 CJK 字体配对，不是简单的颜色替换。尤其是 Chinese Ink 主题，体现了中国文化特色的 AI 视觉设计
3. **人机结对编程实践**：整个项目是在 AI 辅助下完成的，我们探索了一套"人类决策 + AI 执行 + 双重审查"的协作模式

### Q30：你还有什么想补充的？

**答：** 我想强调三点：

**第一，关于隐私。** SlideForge 是桌面应用，用户文档只在本地处理，API Key 通过 Electron safeStorage 加密。在数据隐私越来越受关注的今天，这是一个有社会责任感的设计选择。

**第二，关于可扩展性。** 我们的架构设计是面向扩展的——新增 AI 供应商只需一个 preset 配置，新增主题只需一个 DesignTokens 对象，新增布局类型只需在类型联合中加一个成员并实现对应的渲染逻辑。

**第三，关于用户体验。** 我们从用户角度设计了完整的 Wizard 流程——从主题输入、范围选择、大纲编辑、风格预览到 AI 生成，每一步都有明确的引导和反馈。尤其是"Show Don't Tell"的风格选择，让用户直接看到效果而非读文字描述，这降低了使用门槛。
