## 问题 1：基于 AI 的智能演示文稿——MarkitDown + Banana-Slides？

### 老师潜台词
"市面上已经有类似项目了，你们和 Banana-Slides 有什么区别？是不是套壳？"

### 标准回答

老师好，感谢提出这个问题。Banana-Slides 确实是一个优秀的参考项目，我们前期也调研过它以及 Marp、Slidev、Gamma 等同类工具。我们和它们的核心区别在三个方面：

**第一，产品定位不同。** Banana-Slides 是一个基于 Flask 的 Web 应用，需要部署服务器、依赖 Gemini API，所有文档都要上传到云端处理。而 SlideForge 是一个 **Electron 桌面应用**，所有数据通过 SQLite 存储在本地，文档解析、AI 调用、导出全部在本地完成。对于学生和教师来说，很多论文和课件涉及隐私，不适合上传到第三方服务器，这是我们的核心差异化。

**第二，样式系统不同。** Banana-Slides 通过上传模板图片来参考风格，输出质量高度依赖用户提供的模板。而 SlideForge 内置了 **16 套精心设计的风格预设**，每套都有完整的颜色体系、字体配对、装饰元素和 CSS 变量系统。用户不需要任何设计能力，通过情绪标签就能获得专业级的视觉效果。我们的中国水墨、霓虹赛博、极光等艺术风格是同类工具中没有的。

**第三，交互方式不同。** Banana-Slides 的编辑是传统的 Web UI 操作。SlideForge 提供了 **AI 对话式修改**——用户用自然语言说"把第三页改成双栏对比布局"或"精简第二页的文字"，AI 会自动解析指令并实时更新预览。我们为此设计了完整的 Prompt 工程链路：上下文注入、格式约束、JSON 清洗管道和布局容错映射。

总结来说，Banana-Slides 是一个 Web 端的 AI 幻灯片生成器，而 SlideForge 是一个面向隐私敏感用户、内置设计系统、支持对话式修改的桌面级工具，定位和实现路径都不同。

## 问题 2：巨人肩膀太全，逆袭的突破设计？增量？

### 老师潜台词
"你们用了 Electron、React、AI 这些成熟技术栈，自己的创新点在哪？有没有增量贡献？"

### 标准回答

老师好，这个问题的核心是"增量价值"。我们承认技术栈本身不是创新，创新在于我们如何组合和打磨这些技术来解决一个具体的痛点。我们的增量贡献有三层：

**增量一：双轨样式系统。** 这是我们在调研同类工具后发现的空白——现有工具要么只管预览（如 Slidev），要么只管导出（如 Python-pptx），预览和导出的视觉效果经常不一致。我们设计了 **主进程 StyleConfig + 渲染进程 DesignTokens 的双轨系统**，主进程的 StyleConfig 用于 HTML/PPTX/PDF 三种导出，渲染进程的 DesignTokens 用于实时预览，两者通过统一的 16 套预设保持同步，确保"所见即所出"。这套双轨系统在同类项目中是没有的。

**增量二：Prompt 工程链路。** 我们不是简单地调用 AI API 然后把结果塞进模板。我们设计了一套完整的 **AI 输出解析管道**：将全部幻灯片的标题、正文、布局以 JSON 注入 System Prompt，让 AI 能精确引用每一页；要求 AI 用结构化的 slide-updates 代码块返回修改指令；然后通过多级容错机制解析——先尝试 slide-updates 格式，降级到 json 格式，自动剥离 Markdown 标记，最后对布局名称做容错映射（比如 comparison 自动映射为 two-column）。这个从 Prompt 设计到输出解析的完整链路是我们自己设计的。

**增量三：中国本土化的设计系统。** 我们专门设计了适合中文排版的视觉风格——中国水墨主题使用 Noto Serif SC / Noto Sans SC 字体配对，配合水墨晕染的背景效果；霓虹赛博主题使用赛博朋克色系和网格背景。这些不是简单的颜色替换，而是每套主题都有独立的装饰元素、背景效果、间距体系和组件圆角定义。考虑到目标用户是中国大学生，这种本土化的设计是有实际价值的。

我们不是在巨人肩膀上简单叠加功能，而是在"样式一致性"、"AI 输出可靠性"和"中文设计适配"三个具体问题上做出了自己的解决方案。

## 问题 3：主题和审美？

### 老师潜台词
"你们的 16 套主题风格，审美水准如何？有没有设计理论支撑？"

### 标准回答

老师好，我们的 16 套主题设计分为四个类别：暗色系、亮色系、专业系和艺术系。设计时遵循了三个原则：

**原则一：对比度与可读性优先。** 每套主题的颜色组合都经过对比度验证——暗色主题的正文颜色保证在深色背景上有足够的亮度差，亮色主题保证在浅色背景上有足够的暗度差。标题使用 Display 字体（Noto Serif SC），正文使用 Body 字体（Noto Sans SC），两者在笔画粗细和衬线风格上形成明确的层级对比。

**原则二：情绪标签匹配。** 我们将主题与四种情绪关联——自信（对应 Bold Signal、Electric Studio 等高对比度暗色主题）、激情（对应 Creative Voltage、Neon Cyber 等鲜艳色彩主题）、沉稳（对应 Notebook Tabs、Paper Ink 等低饱和度亮色主题）、灵感（对应 Dark Botanical、Vintage Editorial 等有格调的主题）。用户在创建向导中选择情绪后，系统自动推荐匹配的主题，降低了设计选择的门槛。

**原则三：装饰元素与主题调性统一。** 每套主题不仅定义了颜色和字体，还定义了装饰风格——中国水墨主题使用噪声纹理背景、极简装饰；霓虹赛博使用网格背景和霓虹光效；极光主题使用渐变网格。装饰元素通过 DesignTokens 中的 decorativeStyle 字段控制，确保装饰与主色调不冲突。

坦率地说，我们团队不是专业的设计师，但通过研究 Material Design 3 的设计规范和参考了大量优秀的 PPT 模板，我们的主题系统在色彩搭配和排版层次上达到了"对非设计专业学生来说足够好用"的水平。如果未来有机会和专业设计团队交流，我们可以进一步提升视觉品质。

## 问题 4：视觉设计？

### 老师潜台词
"生成的 PPT 视觉效果怎么样？和市面上的模板比如何？有没有特色？"

### 标准回答

老师好，关于视觉设计，我们有三层考量：

**第一层：布局系统。** 我们实现了 12 种幻灯片布局，从基础的标题页、内容页，到双栏对比、高亮引用、功能网格、大数字展示、时间轴、标注卡片、声明页等。每种布局都有独立的 CSS 响应式规则，通过 clamp() 控制字号和间距，确保在任何屏幕比例上都不会溢出。这意味着 AI 在生成幻灯片时，可以根据内容类型自动选择最合适的布局——数据对比用双栏，流程展示用时间轴，核心观点用高亮。

**第二层：动态预览与导出一致性。** 这是我们在视觉设计上花心思最多的地方。用户在编辑器里看到的效果，必须和最终导出的 HTML/PPTX/PDF 完全一致。为此我们实现了双轨样式系统——主进程的 StyleConfig 负责导出渲染，渲染进程的 DesignTokens 负责实时预览。两者的颜色值、字体配置、装饰参数完全同步，通过 CSS 变量注入到 12 种布局的渲染中。

**第三层：特色视觉风格。** 我们有几个比较有辨识度的主题——中国水墨主题使用灰白底色配合墨色文字和朱砂红点缀，装饰使用 SVG 噪声纹理模拟宣纸质感；霓虹赛博主题使用深蓝黑底色配合青色和品红色霓虹光效，背景使用网格图案。在切换风格时，从水墨切换到赛博，视觉差异非常明显，这个演示效果在现场展示时很能打动人。

当然，我们的视觉设计也有不足——比如目前没有支持用户自定义颜色或上传品牌配色。如果后续迭代，这是我们最想改进的方向。

## 问题 5：三分钟以内——复杂文档比如投行 300-400 页 PPT 能处理吗？会推出并发加速吗？

### 老师潜台词
"你们宣称三分钟生成 PPT，但复杂场景能不能 hold 住？有没有考虑性能优化？"

### 标准回答

老师好，这个问题很关键，我分三层回答。

**第一层：三分钟指的是典型场景。** 我们在宣传中说的"三分钟"是指大学生最常见的使用场景——上传一篇 10-20 页的课程论文或课程笔记，AI 生成 10-15 张幻灯片的 PPT。这个流程包含文档解析（约 5-10 秒）、AI 大纲生成（约 30-60 秒）、AI 幻灯片生成（约 60-90 秒），总计约 2-3 分钟。对于这种规模的文档，我们的性能表现是稳定的。

**第二层：超长文档的技术瓶颈。** 对于投行 300-400 页的行业分析报告，确实会面临两个瓶颈：

一是 **AI 上下文窗口限制**。目前主流的大语言模型（DeepSeek、通义千问等）的上下文窗口在 32K-128K token 之间，一篇 300 页的报告文本量可能达到 100K+ token，单次调用无法处理。我们的解决方案是**文档分片策略**——在文档解析阶段将长文本按章节切分，每次只发送一个章节给 AI 生成对应的幻灯片，然后合并结果。目前我们的代码中已经实现了按 section 分批生成的两阶段策略（Batch1 生成标题页和分隔页，Batch2 生成内容页），这个架构天然支持扩展到更长的文档。

二是 **串行 AI 调用的耗时问题**。假设 300 页报告有 30 个章节、100+ 张幻灯片，逐章生成可能需要 30-50 分钟。这就是老师提到的并发加速问题。

**第三层：关于并发加速。** 我们目前没有推出并发加速服务，但架构上已经预留了扩展空间。具体来说：

- 我们的 AI 适配器模式（`src/main/ai/adapter.ts`）已经是异步的，每个 AI 调用返回 Promise，理论上可以并行发起多个请求。
- 我们的 GeneratingStep 中已经有 Batch1 和 Batch2 的分批策略，要扩展到并发只需要将 `for...of` 串行循环改为 `Promise.all()` 并行调用。
- 瓶颈在于 **API 速率限制**。大多数 AI 服务商对单用户的并发请求数有限制（如 DeepSeek 限制 5 并发、Qwen 限制 10 并发），所以即使我们前端并发了，API 端也会排队。

如果我们未来要做并发加速，方案是：用户在设置中配置多个 API Key（轮询使用），或者部署自己的模型服务（无速率限制），然后前端用 Worker 线程池并发调用。这个方案在技术上是可行的，我们的适配器架构已经支持。

总结：三分钟适用于典型学生场景。超长文档的瓶颈在 AI 上下文窗口和 API 速率限制，不在我们的代码架构。我们的分批生成策略和异步适配器设计已经为未来的并发扩展做好了准备。
