# SlideForge 答辩汇报稿（5 分钟）

> 总字数约 1,100 字，正常语速朗读约 4 分 30 秒，留 30 秒缓冲

## 第 1 页 · 封面（10 秒）

各位老师、同学们好，我们是 SlideForge 项目组，我是胡祖宁，这位是马鑫，这位是王诣博。今天由我来向大家汇报我们小组的期末项目成果——SlideForge，一个 AI 驱动的智能演示文稿生成器。

我们的项目总共编写了 8,500 行源码，设计了 16 套风格模板，并通过了 104 个单元测试。接下来我将从选题背景、技术方案、核心功能和团队协作几个方面进行介绍。

---

## 第 2 页 · 选题背景（30 秒）

我们选择这个课题，是因为发现了大学生制作 PPT 的三大痛点：

第一，**模板千篇一律**。WPS 和 Canva 的模板大家都在用，很容易"撞衫"，缺乏个人风格。

第二，**内容重组困难**。从论文或课程笔记到 PPT，需要大量复制粘贴、重新组织结构，非常耗时。

第三，**迭代修改麻烦**。调整布局、精简文字、更换风格都需要逐页手动修改。

基于这些痛点，我们确定了四个核心目标：上传文档后 AI 自动生成大纲、提供 16 套差异化风格、支持自然语言对话修改幻灯片、以及一键导出三种格式。目标用户主要是大学生和高校教师，使用场景包括期末汇报、学术组会和毕业答辩。

---

## 第 3 页 · 技术选型（25 秒）

技术选型方面，我们使用 **Electron + React 18 + TypeScript + Vite** 作为基础框架。选择 Electron 是因为它支持本地文件读写和 SQLite 数据库，所有数据都在本地处理，不需要服务器。状态管理选择了 Zustand，比 Redux 少了大约 60% 的样板代码。样式方案使用 Emotion CSS-in-JS 实现 16 套风格的动态切换。

项目划分为 8 大功能模块，覆盖从文档上传到导出的完整链路。AI 方面，我们设计了统一的 OpenAI 兼容适配器，通过修改 baseURL 就可以在 GPT-4、Claude、DeepSeek 和通义千问之间无缝切换，业务代码不需要任何改动，并且支持流式 SSE 输出，AI 回复可以逐字显示。

---

## 第 4 页 · 技术架构（25 秒）

架构上我们采用 Electron 标准的三层分离设计——渲染层、预加载层和主进程，通过 IPC 通信解耦。

渲染层负责所有的 UI 交互，包括六步创建向导和三栏编辑器。预加载层作为安全桥梁，暴露类型安全的 API，并处理密钥加密和流式事件监听。主进程负责 AI 调用、三种格式的导出引擎以及 SQLite 数据持久化。

核心功能包括：支持四种文档格式的智能解析、16 套风格模板、12 种幻灯片布局、AI 对话式修改、以及 HTML、PPTX、PDF 三种格式的导出引擎。

---

## 第 5 页 · 核心流程（30 秒）

用户的使用流程分六步：

第一步，**上传文档**，系统支持 PDF、DOCX、TXT、MD 四种格式，自动解析为纯文本。第二步，**AI 生成大纲**，将文档发送给 AI，生成结构化的章节和要点，用户可以在线编辑和排序。第三步，**选择风格**，通过情绪标签推荐匹配的模板，也可以浏览全部 16 套。第四步，**AI 生成幻灯片**，根据大纲和风格自动为每个章节选择最佳布局。第五步，**AI 对话微调**，用户用自然语言说"把第三页改成双栏对比"，AI 自动修改并实时更新预览。最后第六步，**一键导出**。

这里特别介绍一下我们的 Prompt 工程设计：我们将全部幻灯片内容以 JSON 注入 System Prompt，让 AI 能精确引用每一页；同时要求 AI 用 slide-updates 代码块返回结构化 JSON；并设计了自动清洗管道来处理 AI 返回的不规范格式。

大家可以看到，这是我们的三栏编辑器布局——左侧是大纲编辑器，中间是幻灯片实时预览，右侧是 AI 助手聊天区。

我们走一遍创建流程：输入主题、选择范围、选择情绪、浏览风格模板。大家注意看，16 套模板之间的视觉差异非常明显，比如从中国水墨风格切换到霓虹赛博风格。

## 第 7 页 · 团队分工（25 秒）

团队分工方面，三人各负责独立模块，通过 TypeScript 接口契约解耦，工作量基本均衡。

**胡祖宁**负责整体架构设计、Zustand 状态管理和 AI 集成与 Prompt 工程，约占 36%。**马鑫**负责 HTML、PPTX、PDF 三种格式的导出引擎和样式系统，约占 29%。**王诣博**负责 16 套主题设计、12 种布局的渲染器以及 104 个单元测试，约占 35%。

我们的协作模式是"接口契约驱动"：胡祖宁定义 Project、Slide 等核心数据类型，马鑫基于这些类型实现导出引擎，王诣博基于这些类型实现渲染器；同时马鑫定义 StyleConfig 样式接口，王诣博据此实现 DesignTokens 设计系统，确保导出和预览的视觉一致性。

## 第 8 页 · 技术亮点（30 秒）

在开发过程中，我们解决了几个核心的技术难题。

第一是**多模型适配器**，统一了 OpenAI 兼容接口，用户只需在设置里切换 API Key 就能换模型。

第二是**双轨样式系统**，主进程的 StyleConfig 用于导出，渲染进程的 DesignTokens 用于预览，两套系统保持同步，真正做到"所见即所出"。

第三是 **Prompt 工程与 JSON 容错**，AI 返回的格式经常不规范，我们设计了多级容错机制：先尝试 slide-updates 格式，再降级到 json 格式，最后对布局名称做容错映射，比如 comparison 自动映射为 two-column。

第四是**零依赖 HTML 导出**，所有 CSS、JS、字体 URL 都内联在一个文件里，用 clamp() 实现响应式，在任何屏幕上都能正确展示。

---

## 第 9 页 · AI 辅助与总结（25 秒）

在整个开发过程中，AI 辅助了四个维度的工作：架构设计阶段帮助我们对比技术方案、代码调试阶段快速定位问题、测试阶段生成覆盖用例、代码审查阶段检查安全和性能风险。

我们最大的体会是：AI 擅长生成"80 分"的代码，但从 80 分到 100 分的打磨仍然需要人的判断力。关键在于提出好问题，而不是盲目接受生成结果。

最大的技术挑战是 AI 返回 JSON 格式的不稳定性——有时缺少字段、有时用 Markdown 包裹。我们通过多级容错机制解决了这个问题。
